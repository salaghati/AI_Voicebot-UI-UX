## Overview

This directory contains the server-side API surface for the application organized into feature-scoped subdirectories. There are no root-level route files in this directory; instead the functionality is split across multiple focused subdirectories. Each populated subdirectory contains a single TypeScript route implementation (route.ts) that acts as the entry point for HTTP requests for that feature (campaigns, inbounds, kb, workflows). Several other subdirectories exist as organizational containers for settings, reports, and auth-related route implementations and may contain their own route.ts files in their child directories.

Taken together, the subdirectories in this api directory represent the concrete route handlers that respond to API requests for specific features. The campaigns, inbounds, kb, and workflows subdirectories each include a route.ts at their root to implement the primary handlers for those scopes (campaigns also includes an ID-scoped nested directory). The settings, reports, and auth subdirectories are organized as containers for endpoint-specific route handlers and currently contain no root-level files here. This layout lets developers find and update request handling logic on a per-feature basis and keeps related route implementations grouped under clear, feature-named folders.


### File Organization

Files are organized by feature area: each feature (campaigns, inbounds, kb, workflows, etc.) lives in its own subdirectory and provides a route.ts implementation at that scope. Some directories are containers for further endpoint-specific subdirectories (settings, reports, auth) and currently have no root-level files.
